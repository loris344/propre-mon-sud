Conventions

  • slug = URL sans les «/» de bord, «/» internes → «--». Ex : /desinfection-logement/herault/montpellier/ → desinfection-logement--herault--montpellier. La home → slug accueil.
  • handledBy calculé à l'import : existing pour «/» et «/blog/» (routes en dur), blog pour tout «/blog/*» (phase blog dédiée, non implémentée), catchall pour le reste (790 pages, route src/app/[...slug]/page.tsx).
  • Répartition des 839 : 469 « Page ville service », 141 « Page ville débarras type », 67 « Page ville débarras », 42 « Page département service », 39 « Article blog », 11 « Page type débarras », 9 « Réalisation catégorie », 8 « Catégorie blog », 7 « Page persona », + mères/hubs/régionales. Priorités : P1=21, P2=257, P3=364, P4=153, P5=43.
  • Intentions : 677 « Transactionnelle locale », 54 départementale, 27 service, 39 « Informationnelle avec conversion » (blog), etc.
  • Schema annoncé par le plan : 767 pages « LocalBusiness + Service + FAQPage ». L'implémentation N'ÉMET PAS LocalBusiness/FAQPage par page (voir phase 5, décision du 10/06/2026), le champ sert au validateur (présence) et à la règle « FAQ rédigée obligatoire ».

🔬 Détail : algorithme du planning, fichiers générés, résolution d'images

Recalcul du planning (excel-to-seo.py)

Constantes : START_DATE = 2026-06-10 · RAMP_DAYS = 14 · RAMP_PER_DAY = 4 · CRUISE_PER_DAY = 7
Ordre de la file = tri par (Date de publication prévue de l'Excel, puis Priorité, puis ordre de ligne)
position p dans la file → date :
  si p < 14×4 (=56)  : jour = p // 4
  sinon               : jour = 14 + (p-56) // 7
publishAt = START_DATE + jour.  Étalement réel : 10/06/2026 → 08/10/2026 (790 pages datées).

Format de seo-pages.json

{ "meta": { "generatedFrom": "Bondash.xlsx",
            "schedule": { "startDate": "2026-06-10", "rampDays": 14, "rampPerDay": 4, "cruisePerDay": 7 },
            "counts": { "total": 839, "catchall": 790, "blog": 47, "existing": 2 } },
  "pages": [ { id, slug, url, title, type, template, service, departement, ville, region,
               priority, originalPublishAt, publishAt, author, reviewer, keyword,
               secondaryKeywords[], intent, parentUrl, childrenUrls[], requiredLinks[],
               suggestedLinks[], needsImage, imageRequired, imageType, altProposed,
               metaTitle, metaDescription, h1, h2[], faq[], schemaType, canonical,
               noindexBeforePublish, notes, handledBy } ] }
internal-links.json : { "<url source>": [ { target, anchor, type, priority } ] }

Résolution de l'image d'une page (src/lib/seo-images.ts, ordre strict)

  • 1. Image exacte : image-map.json[page.url] si présente.
  • 2. Pool du service : image-fallbacks.json[page.service], choix déterministe hash(page.url) % pool.length (même page = même image, villes voisines = images différentes).
  • 3. Aucun pool → la page se publie sans image. Frontmatter noImage: true force l'absence (cas exceptionnel uniquement, décision client : une image par page).
  • alt : altProposed de l'Excel sinon construit depuis H1 + lieu. Unique par page.
  • Rapprochement dossier photo → service : par tokens normalisés (tolère accents, parenthèses, fautes). Égalité parfaite de score = dossier ignoré avec warning (on n'invente pas).

Procédure de reprise EXACTE pour une nouvelle IA (session vierge)

1. Lire CLAUDE.md (racine du repo) : règles + décisions client. Il PRIME en cas de conflit avec ce document.
2. npm run seo:status  → avancement, retards, prochaine page (et flag ✍︎ description à écrire à la main)
3. Extraire le brief : node -e "const P=require('./src/data/seo-pages.json').pages;
   console.log(JSON.stringify(P.find(p=>p.slug==='<slug>'),null,2))"
4. Si page locale : 1-2 recherches web sur la ville (quartiers, bâti, SCHS/ARS, climat, risques, parc logement).
   Ne JAMAIS écrire un fait local non vérifié. Garder les URLs → frontmatter sources.
5. Écrire content/seo/<slug>.mdx (schéma ci-dessus, règles du tableau, angle du service).

6. npm run seo:validate → corriger jusqu'à zéro erreur (les warnings « lien dormant » sont normaux).
7. Relecture rendue possible : npm run dev → localhost:3000/<url>.
8. Commit + push main. Lots de 10-15 pages max entre checkpoints. JAMAIS deux pages écrites en parallèle.

Validation bloquante (le garde-fou)

AUTOMATIQUE
scripts/validate-seo.mjs contrôle toutes les pages rédigées non-draft. Une règle violée = le déploiement échoue, rien ne part en ligne.

Unicité

  • Meta title unique
  • Meta description unique
  • Similarité corps < 70 %

Complétude

  • H1, canonical https absolu, schema
  • FAQ rédigée, mots minimum
  • Image exacte si exigée

Maillage

  • Liens du corps vers cibles connues
  • Liens « dormants » signalés (tolérés)
✗ ERREUR → build interrompu, rien ne se publie
✓ OK → la machine continue

Logique de publication (src/lib/seo-pages.ts)

buildDate()        = SEO_BUILD_DATE (env, format YYYY-MM-DD, pour tester un jour J) sinon date du jour
publicationDate()  = max(buildDate(), meta.schedule.startDate)   // plancher au 10/06/2026
isPublished(page)  = page.handledBy === "catchall"
                     ET le MDX existe ET draft !== true
                     ET (en dev : toujours vrai | en prod : page.publishAt ≤ publicationDate())
generateStaticParams() ne génère QUE les pages publiées · dynamicParams = false → tout le reste = vrai 404
(out/404.html existe : GitHub Pages le sert avec un statut HTTP 404, pas de soft-404)

Maillage dynamique (tout est recalculé à CHAQUE build)

publicUrlSet = EXISTING_PUBLIC ∪ pages catchall publiées ∪ articles blog publiés (/blog/<slug>/)

1. BLOCS DE MAILLAGE (getMaillage) : liens du plan groupés par type

3. FIL D'ARIANE (buildBreadcrumb) : remonte parentUrl en ne gardant que les ancêtres PUBLIÉS

Élagage trimestriel (stratégie actée, pas encore exécutée)

Cible : pages à 0 impression GSC après 4-6 mois (purge Google documentée vers 90-130 jours pour les pages 0-clic).
Contrainte : GitHub Pages ne fait PAS de 301. Méthode : transformer la page en stub
<meta http-equiv="refresh" content="0;url=/page-departement/"> + canonical vers la cible (équivalent 301 pour Google),
ou enrichir la page si elle a un potentiel. Décision page par page sur export GSC.
